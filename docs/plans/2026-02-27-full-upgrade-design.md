# 全面升级与回测跑通设计

**目标**
- 在不接入自动交易的前提下，形成可重复执行的“升级 + 回测 + 验证”闭环。
- 一键执行后，能产出结构化报告并明确是否“跑通”。

**范围**
- 升级链路：学习视图刷新、实验追踪快照、轻量策略训练、闭环健康检查。
- 回测链路：多策略多标的烟雾回测，输出通过率、失败原因、冠军策略。
- 可观测性：统一写入 JSON/JSONL 报告，便于 UI 与后续 A/B 对比读取。

**通过标准（本轮）**
- 单元测试全绿。
- 升级管线执行无异常中断。
- 回测烟雾测试存在有效样本，且至少 1 个策略-标的组合成功完成回测（不是空数据）。
- 产出 `data/upgrade_pipeline_latest.json` 与 `data/backtest_smoke_latest.json`。

**架构方案（推荐）**
- 方案 A（推荐）：新增统一管线模块（`core/upgrade_pipeline.py`）+ 回测烟雾模块（`core/backtest_smoke.py`）+ CLI 执行器（`tools/full_upgrade_and_backtest.py`）。
- 方案 B：把逻辑散落到 `loop_controller.py` 与 UI 按钮中。
- 方案 C：只人工逐条命令执行，不做代码化。

推荐 A 的原因：
- 可重复、可自动重跑、可外部调度（任务计划/CI）。
- 报告结构稳定，便于后续接 MLflow/DVC。
- 故障定位快（分阶段状态 + 错误明细）。

**数据流**
1. 读取 `event_bus`，刷新学习视图（现有模块）。
2. 读取 `learning_samples + strategy_profiles`，生成实验跟踪快照。
3. 触发轻量策略训练（可配置模式，默认小规模样本，避免超时）。
4. 执行回测烟雾测试并汇总。
5. 生成总体管线报告，写 latest + history。

**错误处理**
- 任一步骤失败不导致整条链路直接崩溃：记录 `ok=false` + `error`，继续后续步骤。
- 回测烟雾把“空数据”和“策略报错”区分统计，避免误判。
- CLI 支持重试（`--max-attempts`），直到通过或达到上限。

**测试策略**
- 先写失败测试（TDD）：
  - 回测烟雾聚合逻辑。
  - 升级管线编排逻辑。
- 再实现最小代码让测试通过。
- 最后跑全量 `unittest` + `py_compile` + 实际管线命令验证。

**执行顺序**
1. 新增测试（先红）。
2. 实现 `backtest_smoke` 与 `upgrade_pipeline`。
3. 新增 CLI 执行器并连接报告落盘。
4. 执行真实“升级+回测”并修复错误，直到跑通。
